#### Unit tests
- Hybrid search: dense + sparse score combination formula; configurable alpha weight
- Cross-encoder reranker: top-K candidates reranked; reranked order differs from original relevance order on known test set
- Knowledge gap detector: query with max-score < threshold → gap event emitted; above threshold → no event
- Multilingual detection: Spanish query correctly detected and language code returned

#### Integration tests (Testcontainers Qdrant + Python pytest)
- **Hybrid search end-to-end:** ingest 3 documents; send BM25-favoring query → hybrid search returns correct document ranked first; pure-dense search returns different ranking
- **Cross-encoder reranker:** ingest 5 documents; top-3 retrieved by dense; cross-encoder reranks → final answer uses cross-encoder top-1 (verified by checking source in response)
- **Min relevance score filter:** configure `min_relevance_score=0.8`; query on unrelated topic → 0 sources returned; response states no information found
- **Configurable top-K:** set `top_k=1` in tenant config → exactly 1 source in response; `top_k=5` → up to 5 sources
- **Multi-turn history:** send 2-message conversation; second message references entity from first → LLM receives rolling context window; answer demonstrates awareness of earlier turn
- **Human escalation webhook:** escalation event reached → `POST` to mock webhook URL with session transcript; `chat_sessions` row updated with `handoff_at`
- **Scheduled re-ingestion:** create re-ingest job for URL; job runs → document re-ingested; old chunks replaced; `updated_at` timestamp advanced
- **A/B test routing:** configure 50/50 split between config A and config B; send 100 requests → ~50 routed to each (chi-squared test on counts); metrics tracked per variant in `ab_test_results` table

## Phase 8 — Retrieval Quality & Advanced Features
Prioritize based on client feedback post-launch.

### Retrieval quality
- [ ] Hybrid search: dense + BM25 (Qdrant sparse vectors)
- [ ] Cross-encoder reranker (`cross-encoder/ms-marco-MiniLM-L-6-v2`, runs locally)
- [ ] Contextual compression (reduce noise in context before LLM call)
- [ ] Configurable top-K and min relevance score threshold per tenant
- [ ] Knowledge gap detection: log queries where max relevance score is below threshold; surface aggregated "unanswered topics" in admin and portal as document improvement suggestions

### Chat experience
- [ ] Multi-turn conversation with history (`chat_messages` as rolling context window)
- [ ] Source citation UI in widget (expandable references below response)
- [ ] Suggested follow-up questions (generated by LLM after each response)
- [ ] Human escalation handoff webhook
- [ ] Multilingual support: auto-detect query language; retrieve in source language, translate context if needed, respond in user's language (LLM-handled or DeepL/Google Translate API)
- [ ] Outbound / proactive messaging: tenant can configure page-URL-pattern or time-on-page rules to auto-trigger widget with a pre-set greeting

### Tenant features
- [ ] Per-tenant model selection (from available LLMs configured on platform)
- [ ] Scheduled re-ingestion (cron: auto-refresh docs from URL or S3 bucket)
- [ ] Webhook subscriptions (per-tenant): `ingestion.complete`, `query.low_confidence`, `escalation.triggered`, `plan.quota_warning` — configurable endpoint + secret per event type (see Phase 10 for full webhook engine)
- [ ] SSO via OIDC (Okta, Entra ID) for enterprise clients
- [ ] Custom domain for widget (`chat.clientdomain.com`)
- [ ] White-label / full branding: custom logo, custom color scheme, remove "Powered by SupportLayer" badge (plan-gated)
- [ ] A/B testing: split live traffic between two bot configurations (system prompt, model, temperature, top-K); compare eval scores and thumbs-up rates, promote winner
- [ ] Email support channel: forwarding address per tenant → RAG-generated draft → agent reviews and sends, or auto-sends above confidence threshold
- [ ] Slack / Microsoft Teams deployment: install bot to an internal workspace channel using the tenant's knowledge base (same `rag-core` pipeline, different surface)

### Platform operations
- [ ] Kubernetes Helm charts for all services
- [ ] Horizontal scaling for `rag-core` (stateless replicas behind load balancer)
- [ ] GDPR: per-tenant data export + full deletion audit trail (Stripe billing integration is Phase 9)
- [ ] Mobile SDK: React Native / WebView wrapper for the embeddable widget (iOS + Android)
- [ ] On-premise / private cloud deployment guide: Helm charts + documented config for air-gapped enterprise deployments
